# Strand Risk Assessment

**Date:** 2026-03-02
**Scope:** Technical, business, and existential risks
**Status:** Comprehensive — synthesized from three parallel risk analyses

---

## Critical Risks (Act Now)

### 1. Solo Founder Distribution Trap

**Likelihood: HIGH | Impact: CRITICAL**

The git log shows high-velocity encoding optimization (v1→v2→v3, 8 experiments, multiple design docs) but zero distribution activity. npm is not published. No blog post written. No Anthropic DevRel outreach. No public visibility.

Every week spent on v4 encoding improvements instead of distribution narrows the window for format standard adoption. This is the classic solo founder trap: satisfying engineering work with clear feedback loops displaces ambiguous distribution work that produces no measurable output for months.

**Mitigation:** Scope lock. Three external milestones:
1. Publish to npm by March 15
2. Publish blog post with experiment data (45→0 tool calls, 78% cost reduction) by March 22
3. Initiate Anthropic DevRel contact by March 31

Encoding improvements are explicitly secondary to these milestones until they are met.

### 2. Anthropic Builds It Natively

**Likelihood: MEDIUM-HIGH | Impact: FATAL**

Anthropic can ship a "project initialization" phase in Claude Code where it runs its own static analysis before the first user turn. This requires approximately one sprint of engineering work. It would not use `.strand` format. It would be whatever internal representation their engineers designed.

**Early warning:** Any Anthropic blog post mentioning "codebase indexing," "project context," or "persistent workspace understanding."

**Mitigation:** Be the acquisition target or the recommended standard before they build their own. The FINDINGS.md data (8 experiments, empirical validation) is the conversation opener. Use it now, not after v4 ships.

### 3. Free Tier Conversion Never Exceeds 2%

**Likelihood: HIGH | Impact: CRITICAL**

The free CLI gives a solo developer everything they need. Industry data: developer tool free-to-paid conversion runs at ~5% median, and the trigger is almost always team adoption (not individual limits). The "paste blast radius in PR → teammate notices → team subscribes" conversion chain has four links at ~40% each = 2.6% chain probability.

**Mitigation:** The free tier needs an architectural boundary at team scale. Options:
- Any repo with >1 contributor requires the App for automated PR comments
- Free generates `.strand` with a "generated by strand free — upgrade for PR automation" footer
- Instrument the activation event aggressively: email the repo owner when the first manual blast radius paste appears in a PR

### 4. Multi-Language Scanner Produces Actively Wrong Output

**Likelihood: HIGH | Impact: HIGH**

The scanner accepts `.py`, `.rs`, `.go`, `.java`, `.rb` files but only understands JS/TS import syntax. Python/Go/Rust files appear as nodes with zero edges. They become "dead code" in the encoding. A Python service in a JS/Python monorepo would show as 47 unreachable files when it is actually the entire backend.

**This is the most likely source of actively wrong output today.**

**Mitigation:** Remove `.py/.rs/.go/.java/.rb` from `isSourceFile` until proper per-language parsers exist. Honest incompleteness is better than confident incorrectness.

## High Risks (Plan For)

### 5. Barrel File Blast Radius Is Wrong in Both Directions

**Likelihood: HIGH | Impact: HIGH**

Barrel files (`index.ts` that re-exports) create two opposing errors:
- **False positives:** The barrel gets 50 importers and massive blast radius, but most importers only use one export and are unaffected by changes to others
- **False negatives:** The actual source file (behind the barrel) shows low blast radius because only the barrel imports it, when the real blast radius is everyone importing through the barrel

These cancel out unpredictably. The blast radius — strand's core differentiator — is systematically unreliable for projects using barrel files (which is most TypeScript projects).

**Mitigation:** Detect barrel files (the scanner already does at lines 177-191) and attenuate blast radius through them. Or resolve barrel re-exports to their actual source files.

### 6. AI Context Windows Will Expand

**Likelihood: HIGH | Impact: HIGH (long-term)**

Gemini 1.5 Pro has 1M tokens. Future models will have larger windows. When an agent can load 500 source files directly, the compression value of `.strand` decreases.

**The counter-argument that saves strand:** Blast radius, churn correlation, and convention detection are pre-computed analysis that is expensive or error-prone for models to compute from raw files. Raw structure information (which files exist, what they export) does not survive the context expansion.

**Mitigation:** Pivot the value proposition from "fits in context" to "pre-computed intelligence." The encoding should contain analysis outputs that are genuinely hard for a model to derive, not structural facts a model can discover by reading files.

### 7. GitHub App Channel Dependency

**Likelihood: MEDIUM | Impact: HIGH**

The GitHub App is the primary paid channel. If GitHub changes webhook policies, limits App permissions, or introduces competing native features, the distribution channel is disrupted. GitHub has changed OAuth and Actions policies before.

**Mitigation:** Don't build exclusively for GitHub. Support GitLab, Bitbucket. Offer a self-hosted option (Docker, GitHub Actions step) where code never leaves the customer's environment. "Zero-egress architecture" is also an enterprise selling point.

### 8. Developer Tool Churn: 5-8% Monthly

**Likelihood: HIGH | Impact: HIGH**

At 8% monthly churn, annual retention is 38%. At 100 customers, you need 8 new customers per month just to stay flat.

**Leading indicator of churn:** Repos that stop regenerating `.strand`. Track "last encoding generation" and email customers whose repos go 30 days without regeneration.

**Mitigation:** Build regeneration into CI/CD on day one (not optional hooks). The Team dashboard ($49 tier) is the retention product — teams that log in to see trends don't churn.

### 9. Per-Repo Pricing Breaks at Enterprise Scale

**Likelihood: HIGH | Impact: MEDIUM-HIGH**

$19/repo × 200 repos = $3,800/month = $45,600/year. That triggers procurement, security review, and legal review. Enterprise will immediately demand a cap or org-level pricing.

**Mitigation:** Plan the pricing architecture before customers arrive:
- Per-org tiers: 1-10 repos ($99/mo), 11-50 repos ($299/mo), 51+ repos ($599/mo)
- Or per-seat pricing (more predictable for customers)
- The $49 Team tier should be the entry point for teams, not an upsell from $19

### 10. Stale .strand Causes Worse Outcomes Than No .strand

**Likelihood: MEDIUM | Impact: HIGH**

Scenario: Developer refactors `auth.ts` into three smaller files. `.strand` still shows the old monolith with `[AMP] amp8.3 ×12→47 d4`. The AI agent adds unnecessary complexity to protect against coupling that no longer exists.

The inverse: A new shared utility file has 30 importers but `.strand` shows it as dead code because it was created after the last generation. The agent doesn't flag it as risky.

**Stale data that produces confident wrong answers is worse than no data.**

**Mitigation:** The freshness system (v4 design) is the right fix. Additionally: `.strand` files older than 7 days should self-identify as stale in the header.

## Medium Risks (Monitor)

### 11. "Complexity" Metric Is Misleading

**Likelihood: HIGH | Impact: MEDIUM**

Complexity is `0.6 * (lines/maxLines) + 0.4 * (imports/maxImports)`. This is relative size-and-coupling, not cognitive complexity. A 300-line type definition file scores higher than a 50-line file with 30 nested conditionals. The word "complexity" in the output creates false confidence.

**Mitigation:** Rename to "weight" or "coupling-size." Or implement actual cyclomatic complexity via AST parsing.

### 12. Dynamic Imports Are Invisible

**Likelihood: HIGH | Impact: HIGH (for affected files)**

`const handler = await import(\`./handlers/${action}\`)` is not caught by the regex. Files loaded only via dynamic imports appear as dead code. An agent suggesting deletion of these files would be wrong.

**Mitigation:** Pattern-match common dynamic import patterns. Flag files in directories commonly used for dynamic loading (handlers/, plugins/, middlewares/) as potentially dynamic.

### 13. Type-Only Imports Create Phantom Edges

**Likelihood: MEDIUM | Impact: LOW-MEDIUM**

`import type { User } from './user'` creates a graph edge but has zero runtime dependency. Type definition files appear riskier than they are because they're widely imported for types only.

**Mitigation:** Detect `import type` separately. Distinguish type-only edges from runtime edges in blast radius computation.

### 14. The .strand Format Has No Versioning Contract

**Likelihood: HIGH | Impact: MEDIUM (over time)**

No schema definition, no documented field names, no parser. Agents read it as free text. Format changes break integrations silently. The CLAUDE.md directive baked into customer repos can't be remotely updated.

**Mitigation:** Publish a format spec. Version the spec (the header already has `v3/v4`). Add a machine-readable section identifier (e.g., `─── RISK` is parseable) and document the contract.

### 15. Open Source Fork Risk (AWS/Elastic Scenario)

**Likelihood: MEDIUM | Impact: HIGH**

A well-capitalized competitor forks the encoder, integrates it natively, and offers it as a free feature. They call it "intelligent codebase analysis."

**Mitigation:** BSL/SSPL for the cloud services. The format spec stays MIT (community adoption). The GitHub App and dashboard are proprietary. This is the HashiCorp model.

### 16. Security: .strand Exposes Architecture

**Likelihood: MEDIUM | Impact: MEDIUM-HIGH**

`.strand` reveals blast radius targets, zero-test-coverage files, domain structure (including "secrets" domain), and churn patterns. For private repos shared with contractors or made partially public, this is an intelligence disclosure.

**Mitigation:** Add guidance to `strand init` about public vs private repos. Consider a `--redact` flag that strips file paths to relative names only.

### 17. npm Supply Chain Risk

**Likelihood: LOW per-package | Impact: CRITICAL (reputational)**

If a transitive dependency is compromised, strand ships malware to developer machines.

**Mitigation:** Lock dependency tree. Use `npm audit` in CI. Publish with `--provenance`. Keep transitive dependencies minimal.

### 18. Git Shallow Clone Kills Churn

**Likelihood: HIGH in CI | Impact: MEDIUM**

GitHub Actions default checkout is `--depth=1`. `git log --since="30 days ago"` returns nothing. Churn data silently disappears. The CHURN section is empty with no warning.

**Mitigation:** Detect shallow clone and warn. Or fail gracefully with `[CHURN unavailable — shallow clone detected]` in the output.

### 19. Monorepo Cross-Package Edges Are Missing

**Likelihood: HIGH for monorepo users | Impact: HIGH**

In Turborepo/Nx monorepos, `import { Button } from '@company/ui'` looks like an external package import. It's dropped from the graph. Cross-package blast radius is invisible.

**Mitigation:** Read workspace `package.json` files. Resolve workspace package imports to their actual source. This is a significant scanner enhancement.

### 20. 10,000-File Codebase Performance Breakdown

**Likelihood: HIGH for large projects | Impact: HIGH**

`walkDir` reads every file with `readFileSync`. BFS blast radius is O(nodes × edges). `git log` has a hardcoded 10MB buffer. At 10K files: 30-120 seconds (breaks the "5-second promise"). At 100K files: crashes (V8 heap limit).

**Mitigation:** Add a file count limit with a scoping flag (`--include src/`). Make BFS lazy (only compute for top-N files by direct importer count). Switch git to streaming spawn.

## The Three Sacred Cows to Kill

**1. "Complexity" is not complexity.** It is relative size-and-coupling. The label creates false confidence. Rename it or implement real cyclomatic complexity.

**2. Multi-language file extensions are a liability.** `.py`, `.rs`, `.go` files are accepted but not understood. They produce ghost nodes that pollute dead code detection. Remove them until proper parsers exist.

**3. The 30-day churn window silently fails on shallow clones.** Every CI/CD system using `--depth=1` produces empty churn data. Since churn is a key differentiator, this silent failure in the most common automated environment is a critical gap.

## The Meta-Risk

**The strategy is sequentially fragile.** Format adoption depends on platform partnerships. Partnerships depend on distribution traction. Traction depends on publishing to npm and writing blog posts. None of those have happened yet.

The single highest-leverage action right now is not v4. It is publishing the FINDINGS.md research as a blog post with the specific numbers: 45 tool calls → 0, 3 minutes → 1.3 minutes, 78% cost reduction with model downgrade. Those numbers are better than any marketing copy. They generate more adoption signal in two days than a month of encoding improvements.

**The risk is that you keep building a better hammer while the nail remains undriven.**
