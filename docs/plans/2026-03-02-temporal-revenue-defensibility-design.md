# Temporal Revenue & Defensibility Design

**Date:** 2026-03-02
**Status:** Approved
**Goal:** Design a revenue model that survives platform absorption, targeting $5-7K MRR to enable full-time work on Strand.

---

## Problem Statement

Strand's current `.strand` file is a static artifact that any platform (Anthropic, Cursor, GitHub) could replicate in one sprint. The current business strategy identifies speed, format standards, and licensing as moats — but industry evidence (HashiCorp, Elastic, Windsurf) shows these are not durable defenses.

**Constraint:** Solo founder with full-time job + restaurant. Cannot outship platforms on speed. Cannot maintain cloud infrastructure initially. Must reach $5-7K MRR to go full-time.

## Core Insight

Platforms can replicate point-in-time codebase analysis. They **cannot** retroactively generate 6 months of architecture evolution from git history. The temporal dimension is the defensible moat.

Strand's scanner already reads the filesystem and builds a full dependency graph. By checking out prior git commits and running the scanner at each checkpoint, Strand can reconstruct weekly architecture snapshots — giving users "6 months of architecture trends" on day one of install.

## Product Architecture

### Free Tier (CLI — distribution engine)

What exists today, plus npm publication:
- `.strand` file generation (scanner + analyzer + encoder)
- Point-in-time analysis: blast radius, complexity, conventions
- Local MCP server for AI assistants
- `strand impact <file>` command
- `strand validate-plan` command

### Pro Tier (temporal intelligence — the product)

New commands gated behind license key:
- `strand history` — reconstruct weekly architecture snapshots from git history
  - Walks git commits (one per week, configurable depth)
  - At each checkpoint: checkout commit, run scanner, store snapshot
  - Stores snapshots in `.strand-history/` directory (local, version-controllable)
  - Default: 26 weeks (6 months) of weekly snapshots
- `strand trend <file-or-module>` — show how metrics changed over time
  - Blast radius growth/shrinkage
  - Complexity trends
  - Import count changes
- `strand drift` — detect convention and coupling drift over time
  - Which files broke conventions and when
  - Which module boundaries are eroding
  - Coupling growth between domains
- Output formats: terminal (default), markdown, JSON

### Team Tier (workflow integration — the retention engine)

- GitHub App (thin layer): automated PR comments with historical context
  - "This file's blast radius grew 40% this quarter — review carefully"
  - "This PR adds a new cross-domain dependency (encoder → cli) — intentional?"
- Slack/webhook integration: architecture anomaly alerts
  - "Coupling between modules X and Y increased 3 consecutive weeks"
  - "Test coverage for payments domain dropped below 60%"
- Team dashboard: static site generated from `.strand-history/` data
  - Architecture health over time
  - Hotspot trends
  - Convention adherence scores

### Enterprise Tier

- Multi-repo cross-dependency analysis
- Architecture health scoring with quarterly reports
- Priority support / SLA

## Defensibility Analysis

### What platforms CAN replicate (not defensible)

- Point-in-time dependency analysis
- Complexity scoring
- Single-snapshot blast radius
- Convention detection

### What platforms CANNOT easily replicate (defensible)

- **Temporal reconstruction from git history**: requires deliberate engineering investment platforms won't prioritize (they're focused on model quality, not architecture history)
- **Historical trend data**: once accumulated, cannot be regenerated by a competitor starting fresh
- **Cross-repo dependency mapping**: platforms operate within single-repo context
- **Architecture drift detection**: requires baseline + history to detect deviation

### Why the moat compounds

1. **Month 1**: user gets 6-month backfill — immediate value
2. **Month 3**: real-time snapshots supplement backfill — data quality improves
3. **Month 6**: team has configured alerts, trained on thresholds — workflow embedded
4. **Month 12**: switching means losing a year of architecture trends — high switching cost

## Key Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Backfill too slow on large repos (10K+ files × 26 checkpoints) | High | Medium | Parallel processing, skip unchanged modules, configurable depth |
| License key easily bypassed for CLI tool | Medium | Medium | Honor system for indie devs, enforce for teams via GitHub App |
| Teams don't see enough value in trends alone | Medium | High | PR risk comments (Phase 2) are the "aha" moment — accelerate if needed |
| Anthropic ships native temporal analysis | Low | Critical | Their focus is model quality, not architecture tooling. Ship first. |
| Git history backfill produces inaccurate snapshots for old commits | Medium | Medium | Validate against known architecture changes, surface confidence scores |
