## Decision 1: Pricing — Per-Repo at Launch, Per-Org at Scale

**Decision:** Launch with per-repo pricing. Add per-org tiers when any single customer reaches 20+ repos.

**What we're shipping:**

| Tier | Price | Trigger to build |
|------|-------|-----------------|
| Free | $0 | Now |
| Pro | $19/repo/month | GitHub App launch (Month 4-6) |
| Team | $49/repo/month | First team customer asks for dashboards |
| Org | $99-599/org/month (tiered) | First customer hits 20+ repos |
| Enterprise | Custom | First inbound enterprise inquiry |

**Rationale:** Per-repo is the simplest model to launch with and the easiest for a 3-person team to say yes to without procurement. The risk doc is correct that it breaks at 200 repos, but that's a problem we solve with revenue, not before it. Building org-tier pricing before having paying customers is premature optimization.

**What we rejected:**
- Per-seat pricing. Repos are the natural unit for a codebase analysis tool. Per-seat creates misaligned incentives (teams avoid adding seats) and is harder to self-serve.
- Making Team ($49) the entry point instead of Pro ($19). The $19 price point lets a single developer expense it without manager approval. That's the wedge. Teams upgrade to $49 when they want the dashboard — the upsell is natural, not forced.

## Decision 2: Free Tier — Generous CLI, Instrumented Conversion

**Decision:** Never gate the CLI. Instead, instrument the conversion event and follow up manually.

**What this means:**
- `strand generate`, `strand impact`, `strand validate-plan`, local MCP server — all permanently free, no contributor limits, no repo size limits, no footers.
- The free tier is the distribution engine. Gating it kills adoption before the format standard has a chance.

**What we're adding from the risk doc:**
- **Instrument the "blast radius paste" event.** When a user runs `strand impact` and the output appears in a GitHub PR comment (detectable via GitHub App webhooks once installed), flag that repo as activation-ready.
- **Opt-in telemetry** on CLI usage (repo size, section usage, regeneration frequency). This feeds the conversion funnel without gating the product.
- **No "upgrade" footers** in `.strand` output. The file is a format standard play — branding it as freemium undermines credibility.

**What we rejected:**
- Requiring the GitHub App for multi-contributor repos. This gates the exact users most likely to evangelize the tool to their team.
- Adding "generated by strand free" footers. This pollutes the format and signals "free tool" rather than "industry standard."
- Email-on-first-paste automation. Too aggressive for early stage. Manual follow-up is fine until there are more than 50 activated repos.

**Risk we're accepting:**
- Conversion may land at 2-3% instead of 5%. That's fine. At 2.5% conversion with 20K free users, that's 500 paying repos = ~$9,500 MRR at blended pricing. Enough to sustain.

## Decision 3: GTM — Distribution First, Product Second

**Decision:** Adopt the risk doc's milestone deadlines. Encoding improvements are frozen until all three are met.

**The milestones:**

| # | Milestone | Deadline | Status |
|---|-----------|----------|--------|
| 1 | Publish to npm (`npx strand` works globally) | March 15 | Not started |
| 2 | Publish blog post with experiment data | March 22 | Not started |
| 3 | Initiate Anthropic DevRel contact | March 31 | Not started |

**Scope lock:** No v4 encoding work, no new scanner features, no experiment runs until milestones 1 and 2 are complete. Milestone 3 can run in parallel.

**Blog post must include:**
- 45 tool calls -> 0 (Experiment 7)
- 3 minutes -> 1.3 minutes real-world comparison
- 78% cost reduction with model downgrade (Experiment 6)
- 96% self-documenting format (Experiment 8)

**Rationale:** The risk doc nailed this. Eight experiments and zero public presence is a distribution failure. The data is better than any marketing copy. Every week it sits in FINDINGS.md instead of a blog post is wasted leverage. v4 improvements don't matter if nobody knows v3 exists.

**What we rejected:**
- The strategy doc's ordering (ship v4 -> npm -> blog). v4 is a nice-to-have. Distribution is existential.
- Delaying npm publish until the CLI is "polished." Ship what works. Iterate publicly.

## Decision 4: Value Proposition — Lead with Intelligence, Not Compression

**Decision:** Reframe the pitch from "fits in context" to "pre-computed codebase intelligence."

**Why:** The risk doc is right that context windows will expand. Compression is a depreciating asset. Blast radius, churn correlation, and convention detection are *analysis outputs* — expensive for models to compute even with unlimited context. That's the durable value.

**One-liner update:**
- Old: "Strand gives AI coding agents a map of your codebase so they stop exploring and start working."
- New: "Strand pre-computes the codebase intelligence that AI agents can't derive from reading files — blast radius, risk amplification, and architectural topology."

**This affects:**
- Blog post framing
- npm README
- Any future pitch to Anthropic DevRel
